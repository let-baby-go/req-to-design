## 📐 第一部分：资深架构师设计标准

### 一、业务流程设计标准

**必须考虑**（缺少即技能执行失败）:
- ✅ 正常流程 + 异常流程（至少 3 种异常场景）
- ✅ 前置条件校验（权限、状态、数据有效性）
- ✅ 后置条件处理（状态变更、消息通知、数据同步）
- ✅ 并发场景（重复提交、并发修改、资源竞争）
- ✅ 边界条件（数据量为 0、极大值、超长文本）

**字段流转分析**（强制执行）：
> 业务流程设计必须清晰分析每个环节的**数据字段流转**和**产生时间**，指导后续的接口设计和数据库设计

- 必须识别业务流程的每个环节的输入字段/输出字段
- 必须分析字段来源（从哪个环节来）和流向（到哪个环节去）
- 必须区分持久化字段和临时字段
- 输出：业务字段流转表（环节/字段名/类型/来源/流向/是否持久化/说明）

**输出要求**:
- 流程图（文字描述或图形）
- 状态机（状态定义、流转条件、触发动作）
- 时序图（多角色/多系统交互）
- 业务字段流转表（字段级别分析）

### 二、核心逻辑设计标准（替换原"架构设计"）

**⚠️ 重要原则**：
- ❌ 禁止展示分层架构图（Controller→Service→Mapper）- 这是形式化内容
- ❌ 禁止简单罗列类名和方法名 - 没有实际价值
- ✅ 必须展示业务逻辑如何在类之间流转
- ✅ 必须说明核心类的职责和协作关系
- ✅ 必须说明业务规则的实现位置

    ↓ 调用

    ↓ 测试连接

| 业务规则 | 实现位置 | 实现方式 |

| 密码加密 | Sm4Util.encrypt() | SM4 算法加密后存储 |

| 异常类型 | 捕获位置 | 处理方式 |
|---------|---------|---------|

### 二、接口设计标准

**必须遵循**（缺少即技能执行失败）:
- ✅ RESTful 规范 - 资源命名、HTTP 方法、状态码
- ✅ 幂等性设计 - GET/PUT/DELETE 必须幂等，POST 考虑防重
- ✅ 版本控制 - URL 或 Header 包含版本号
- ✅ 错误码体系 - 统一错误码格式、含义清晰、便于排查
- ✅ 参数校验 - 必填/可选、格式校验、长度限制、业务规则
- ✅ 响应规范 - 统一返回格式、分页参数、排序参数
- ✅ 安全考虑 - 鉴权、限流、敏感数据脱敏
- ✅ **复用判断标准** - 基于业务语义匹配，不是名称相似；必须读取源码分析业务逻辑；根据复用判断矩阵客观评分
  - ❌ **禁止为了复用而复用** - 不强行复用不匹配的接口

#### 接口调用方分析与设计标准（强制执行）

**AI 必须先判断接口调用方，再设计接口**。不同调用方的设计要求完全不同：

**1. 调用方类型识别**：

| 调用方类型 | 识别特征 | 设计要点 | 典型场景 |
|-----------|---------|---------|---------|
| **页面调用**（前端→后端） | - 用户通过浏览器访问<br>- 有登录态/会话管理<br>- 权限控制到按钮级别 | - 用户会话管理<br>- CSRF 防护<br>- 防重复提交<br>- 按钮权限控制 | 管理后台、C 端页面 |
| **对外 API 调用**（系统→系统） | - 第三方系统调用<br>- 无用户会话<br>- 需要 API 密钥 | - API 网关<br>- 签名验签<br>- 限流熔断<br>- 请求日志 | 开放平台、合作伙伴对接 |
| **内部服务调用**（服务→服务） | - 微服务间调用<br>- 内网通信<br>- 高性能要求 | - 服务发现<br>- 负载均衡<br>- 超时重试<br>- 熔断降级 | 微服务架构内部调用 |
| **定时任务调用**（系统自动） | - 系统自动触发<br>- 批量处理<br>- 后台执行 | - 分布式锁<br>- 幂等性<br>- 失败重试<br>- 进度监控 | 数据同步、定时对账 |

**2. 鉴权设计标准**：

**页面调用鉴权**：
```markdown
**必须设计以下内容**：
1. **用户认证**：
   - 登录态：Session/Cookie/JWT（选择一种并说明理由）
   - 有效期：通常 30 分钟 -24 小时
   - 刷新机制：自动刷新/手动刷新

2. **权限校验**：
   - 功能权限：菜单/按钮级别（如：只有管理员能看到"删除"按钮）
   - 数据权限：部门/角色/个人数据范围（如：只能看本部门数据）
   - 操作权限：增删改查权限分离（如：只读角色不能调用 POST 接口）

3. **安全措施**：
   - CSRF Token 防护（表单提交必须）
   - XSS 防护（输入过滤、输出转义）
   - 敏感操作二次验证（如删除、转账）
```

**对外 API 调用鉴权**：
```markdown
**必须设计以下内容**：
1. **认证方式**（三选一，说明理由）：
   - API Key/Secret：HMAC-SHA256 签名
   - OAuth2.0：Access Token + Refresh Token
   - 双向 SSL：证书认证

2. **签名算法**（必须完整描述）：
   ```
   签名生成步骤：
   1. 将请求参数按字典序排序
   2. 拼接：参数串 + 时间戳 + nonce
   3. 计算：HMAC-SHA256(拼接串，Secret)
   4. 将签名放入请求头：X-Signature: {signature}
   
   验签步骤：
   1. 服务器用相同算法计算签名
   2. 对比客户端签名与服务器签名是否一致
   3. 不一致则返回 401 Unauthorized
   ```

3. **防重放攻击**：
   - 时间戳校验（允许±5 分钟误差）
   - Nonce 唯一性校验（Redis 缓存，有效期 5 分钟）

4. **限流策略**（必须说明具体数值）：
   - 按 API Key 限流：如 1000 次/分钟
   - 按 IP 限流：如 100 次/分钟
   - 分级限流：普通用户 1000 次/天，VIP 用户 10000 次/天
```

**内部服务调用鉴权**：
```markdown
**必须设计以下内容**：
1. **认证方式**：
   - 内网隔离（网络层隔离）
   - 服务间 Token（如 JWT，有效期较长）
   - mTLS（双向 SSL 认证）

2. **超时控制**：
   - 连接超时：如 3 秒
   - 读取超时：如 10 秒
   - 重试次数：如 3 次，间隔递增（1s, 2s, 4s）

3. **熔断降级**：
   - 熔断条件：如 5 秒内失败率>50%
   - 降级方案：返回默认值/缓存数据/错误提示
```

**3. 并发控制设计标准**：

| 并发场景 | 解决方案 | 适用情况 | 设计要点 |
|---------|---------|---------|---------|
| **重复提交** | 前端按钮禁用 + 后端幂等性（Token 机制） | 表单提交 | 1. 提交后按钮禁用<br>2. 生成提交 Token，提交后失效<br>3. 重复提交返回"请勿重复提交" |
| **并发修改** | 乐观锁（version 字段）/ 悲观锁（SELECT FOR UPDATE） | 数据更新 | 1. 乐观锁：更新时检查 version，失败则提示"数据已变更"<br>2. 悲观锁：加锁读取，更新后释放，注意死锁风险 |
| **资源竞争** | 分布式锁（Redisson）/ 队列串行化 | 库存扣减、名额限制 | 1. 分布式锁：lock(key, timeout)<br>2. 队列：请求入队，单线程消费<br>3. 超时释放，防止死锁 |
| **高并发读** | 缓存（Redis）+ 读写分离 | 商品详情、配置查询 | 1. 缓存策略：先查缓存，无则查库并回写<br>2. 缓存失效：主动失效/TTL<br>3. 缓存穿透：空值缓存/布隆过滤器 |
| **高并发写** | 消息队列削峰 + 批量处理 | 日志记录、统计数据 | 1. 请求入队，立即返回<br>2. 后台消费，批量写入<br>3. 失败重试，告警通知 |

**接口文档 9 要素**（缺一不可）:
1. 接口路径 + 方法 + 版本
2. 接口名称 + 功能描述
3. 请求参数（名称、类型、必填、校验规则、示例）

9. 性能要求（QPS、响应时间、并发量）

**详细程度要求**（强制执行）：
> **详细到用户拿到设计文档就能直接写代码**

- **请求参数表**: 每个接口必须有完整的请求参数表格（参数名、类型、必填、说明）
- **响应参数表**: 每个接口必须有完整的响应参数表格（字段名、类型、说明、示例值）
- **不同类型分别设计**: 如果请求参数因类型不同而不同（如 config），必须为每种类型设计完整的字段表格，不能用 Object 笼统带过
- **流程图**: 每个接口必须有完整的业务流程图（参数校验 → 业务处理 → 异常处理）
- **伪代码**: 复杂业务逻辑需要提供关键逻辑的伪代码
- **⚠️ 所有接口必须完整设计，不能省略任何一个接口**：无论有多少个接口，每个接口都要按上述要求完整设计，不能用"按同样格式设计"、"略"、"..."、"接口 6-18"等方式省略

**接口设计特殊要求**（如适用，强制执行）:
- 如有**多种配置类型**（如 6 种数据源）→ 每种类型的 config 字段必须单独列表
- 如有**复杂参数校验** → 必须说明校验规则和失败处理

### 核心逻辑 1：双方签署同一版本后才备案

**识别来源**：
- 需求依据：MD 第 3.3 章"数字合约管理" - "合约签署（使用主体私钥签名）"
- 4.1 流程：流程 1 第 18 步"双方都签署 v1 后备案"
- 业务问题：需方和供方可能签署不同版本，需确保双方对同一版本签名后才备案

**设计要点**：
1. **核心机制**：
   - 版本控制：每次修改 version+1
   - 签署状态独立：creator_sign_status / receiver_sign_status
   - 备案条件：creator_sign=1 AND receiver_sign=1 AND 同一 version

2. **关键实现位置**：
   - `ContractService.sign()` - 签署逻辑
   - `ContractService.checkFilingCondition()` - 备案条件检查
   - `ContractFilingTask.execute()` - 定时扫描备案

3. **依赖的公共组件**：
   - `initiatorAuthUtil.doBidirectionalAuth()` - 双向身份认证
   - `SignatureUtils.sign()` - 使用主体私钥签名
   - `SignatureUtils.verify()` - 验签

4. **异常处理**：
   - 验签失败：返回"签名无效"，状态不变
   - 版本不一致：返回"版本已变更，需重新签署"
   - 备案失败：filing_success=0，定时任务重试

5. **性能考虑**：
   - QPS 预估：100 次/分钟（签署操作低频）
   - 缓存策略：合约详情缓存 5 分钟
   - 限流策略：单用户 10 次/分钟（防止恶意签署）

**实现伪代码**：
```java
// ContractService.sign()
public void sign(Long contractId, Integer version, String signature) {
    // 1. 双向身份认证
    initiatorAuthUtil.doBidirectionalAuth();
    
    // 2. 获取合约
    Contract contract = contractMapper.selectById(contractId);
    
    // 3. 版本校验
    if (!version.equals(contract.getVersion())) {
        throw new BusinessException("版本已变更，当前版本：" + contract.getVersion());
    }
    
    // 4. 验签
    boolean valid = SignatureUtils.verify(contract.getContent(), signature, currentUser.getPublicKey());
    if (!valid) {
        throw new BusinessException("签名无效");
    }
    
    // 5. 更新签署状态
    if (currentUser.isCreator()) {
        contract.setCreatorSignStatus(1);
        contract.setCreatorSignature(signature);
    } else {
        contract.setReceiverSignStatus(1);
        contract.setReceiverSignature(signature);
    }
    
    // 6. 检查备案条件
    if (contract.getCreatorSignStatus() == 1 
        && contract.getReceiverSignStatus() == 1) {
        contract.setStatus(2); // 已生效
        contract.setFilingTime(new Date());
    }
    
    contractMapper.update(contract);
}
```

  - `POST /contract/negotiate` - 协商接口（复用双向认证逻辑）
  - `GET /contract/detail` - 详情接口（复用查询逻辑）
- 需新增的接口：
  - `POST /contract/sign` - 签署接口（新增签署逻辑）
  - `POST /contract/file` - 备案接口（新增备案推送）

**数据流转说明**：
- 输入字段：
  - contractId、version、signature → 来自前端签署操作
  - 当前用户信息 → 来自双向身份认证
- 输出字段：
  - creator_sign_status、receiver_sign_status → 持久化到 t_contract 表
  - status → 持久化到 t_contract 表
  - filing_time → 持久化到 t_contract 表
- 持久化字段：
  - t_contract.creator_sign_status (INTEGER)
  - t_contract.receiver_sign_status (INTEGER)
  - t_contract.status (INTEGER)
  - t_contract.filing_time (DATETIME)

### 五、可插拔规则引擎设计标准（如适用）

**判断标准**（满足 2 项以上建议采用规则引擎）：
- □ 业务规则≥3 种，且规则可能变化（如：次数限制、日期限制、时间段限制）
- □ 规则需要动态配置（无需改代码，通过配置新增/修改规则）
- □ 规则需要支持扩展（未来可能新增规则类型）
- □ 多个模块共用同一套规则框架（如：交付订单、合约管理都用规则引擎）
- □ 规则校验逻辑复杂（多条件组合、规则优先级、规则链）

**不需要的情况**（满足任意一项即不建议采用）：
- □ 规则固定不变（硬编码即可，如：只有 1 种校验逻辑）
- □ 只有 1-2 种简单规则（if-else 足够）
- □ 规则逻辑极其简单（如：只校验一个条件）
- □ 性能要求极高（规则引擎有额外开销）

**如不需要规则引擎**：跳过本节，说明理由（如："只有 1 种校验逻辑，直接硬编码"）

# 三遍校验机制（强制执行）

**版本**：v1.0  
**最后更新**：2026-04-18  
**适用范围**：req-to-design 技能第 4 章详细设计

---

## 🔄 什么是三遍校验

**核心原则**：每个设计输出后，必须反复对照需求文档校验三遍，确保设计能完整闭环

**校验目标**：
- ✅ 设计内容能追溯到需求文档的具体章节
- ✅ 需求文档的每个功能点都有对应的设计
- ✅ 接口设计能覆盖业务流程的所有环节
- ✅ 数据库设计能支撑接口的所有字段

**去掉形式化内容**：
- ❌ 分层架构图（Controller→Service→Mapper）- 没有实际价值
- ❌ 简单罗列类名 - 要展示业务逻辑流转
- ❌ 套话空话 - 只写具体的业务逻辑

---

## 📝 第一遍校验：业务流程设计完成后

**校验时机**：4.1 业务流程设计输出后

**校验步骤**：

### 第一遍：对照需求文档
```
1. 打开需求文档的对应章节（如 3.2.1 数据源接入）
2. 逐段阅读需求描述
3. 检查业务流程是否覆盖了需求的每个功能点
4. 标记未覆盖的需求点
```

**检查清单**：
- [ ] 需求文档的每个功能点是否都有流程对应？
- [ ] 异常场景是否至少 3 种？
- [ ] 字段流转是否清晰（从哪个环节来到哪个环节去）？
- [ ] 状态流转是否有触发条件和守卫条件？

### 第二遍：对照接口设计
```
1. 打开接口设计文档
2. 检查业务流程的每个环节是否有对应接口
3. 检查接口的参数是否来自流程的输入字段
4. 检查接口的响应是否对应流程的输出字段
```

**检查清单**：
- [ ] 流程的每个环节是否有接口对应？
- [ ] 接口参数是否来自流程输入？
- [ ] 接口响应是否对应流程输出？
- [ ] 是否有接口无法解释的流程环节？

### 第三遍：对照数据库设计
```
1. 打开数据库设计文档
2. 检查接口的每个字段是否有表字段存储
3. 检查表的字段是否能支撑所有接口
4. 检查是否有字段无法解释来源
```

**检查清单**：
- [ ] 接口字段是否有表字段存储？
- [ ] 表的字段是否都能被接口使用？
- [ ] 是否有字段无法解释来源？
- [ ] 是否有接口需要的字段表中没有？

---

## 📝 第二遍校验：核心逻辑设计完成后

**校验时机**：4.2 核心逻辑设计输出后

**⚠️ 核心逻辑设计不是分层架构，而是业务逻辑流转**

**必须展示**：
- ✅ 核心类的职责和协作关系
- ✅ 关键方法的调用链路
- ✅ 业务规则的实现位置
- ✅ 异常处理的触发点

**禁止展示**：
- ❌ Controller→Service→Mapper 分层图
- ❌ 简单罗列类名和方法名
- ❌ 没有业务含义的技术架构

**校验步骤**：

### 第一遍：对照需求文档
```
1. 打开需求文档的对应章节
2. 检查核心逻辑是否实现了需求的所有业务规则
3. 检查是否有需求未实现的逻辑
```

**检查清单**：
- [ ] 需求的每个业务规则是否有逻辑实现？
- [ ] 业务规则的实现位置是否清晰？
- [ ] 异常处理是否覆盖了需求的异常场景？

### 第二遍：对照业务流程
```
1. 打开业务流程设计
2. 检查核心逻辑是否实现了流程的每个环节
3. 检查流程的流转是否有代码逻辑支撑
```

**检查清单**：
- [ ] 流程的每个环节是否有逻辑实现？
- [ ] 流程的流转是否有代码支撑？
- [ ] 状态流转是否有对应的逻辑？

### 第三遍：对照接口设计
```
1. 打开接口设计文档
2. 检查接口的处理逻辑是否有核心类实现
3. 检查接口的参数校验是否有逻辑支撑
```

**检查清单**：
- [ ] 接口的处理逻辑是否有核心类？
- [ ] 参数校验是否有逻辑支撑？
- [ ] 接口异常是否有处理逻辑？

## 📝 模块级三遍校验（最终校验）

**校验时机**：一个模块的所有设计完成后

**目的**：确保整个模块的设计能完整闭环

### 第一遍：需求→设计追溯
```
1. 打开需求文档的对应章节
2. 逐段阅读需求
3. 为每个需求点标记对应的设计文档章节
4. 检查是否有需求没有设计对应
```

**输出**：需求追溯表
| 需求章节 | 需求描述 | 业务流程 | 核心逻辑 | 接口 | 数据库表 |
|---------|---------|---------|---------|------|---------|
| 3.2.1.1 | 新增数据源 | 4.1 第 2 章 | 4.2 第 3 章 | 接口 2 | edb_datasource |
| 3.2.1.2 | 数据源列表 | 4.1 第 4 章 | 4.2 第 3 章 | 接口 4 | edb_datasource |

### 第二遍：设计→需求回溯
```
1. 打开所有设计文档
2. 逐个检查每个设计内容
3. 为每个设计内容标记对应的需求章节
4. 检查是否有设计没有需求依据
```

**输出**：设计回溯表
| 设计内容 | 设计描述 | 对应需求 | 是否必需 |
|---------|---------|---------|---------|
| 接口 2 | 创建数据源 | 3.2.1.1 | ✅ |
| 接口 X | XXX | 无 | ❌ 删除 |

### 第三遍：完整性校验
```
1. 对照需求追溯表和设计回溯表
2. 检查是否所有需求都有设计
3. 检查是否所有设计都有需求
4. 检查是否有遗漏或冗余
```

**输出**：完整性报告
```
【完整性校验结果】

需求覆盖度：100%
- 需求章节数：15
- 有设计对应的需求：15
- 无设计对应的需求：0

设计必要度：100%
- 设计内容数：20
- 有需求依据的设计：20
- 无需求依据的设计：0

【校验结论】
✅ 设计完整闭环，可以进入下一个模块
```

## 📊 校验质量评分

**每个模块完成后，AI 必须自评**：

| 评分项 | 分值 | 评分标准 |
|-------|------|---------|
| **需求覆盖度** | 25 分 | 100%=25 分，每少 5% 扣 5 分 |
| **流程完整度** | 25 分 | 所有环节有接口=25 分，每少 1 个扣 5 分 |
| **接口完整度** | 25 分 | 所有参数有字段=25 分，每少 1 个扣 5 分 |
| **数据库完整度** | 25 分 | 所有字段被使用=25 分，每多 1 个冗余扣 5 分 |

**通过标准**：≥85 分才能进入下一个模块
